Private peer-to-peer messaging on Elastos

Hey Chat × Hey Core

A thin chat UI over a shared, post-quantum, sealed-sender messaging engine — talking device-to-device over the Carrier P2P network.

No central server holds your messages. No relay sees who you are. The crypto core is shared byte-for-byte with Hey Social — one network, one audit surface.

The stack

One engine, two apps, four layers

hey-chat is just UI. Every secret operation lives in hey-core, compiled into the app's WebAssembly.

Because both apps link the same hey-core, they're byte-identical on the wire — they interoperate as one chat network, and there's only one copy of the dangerous code to audit.

The auth part

Your identity lives in the runtime

Hey apps are wallet capsules: the Elastos runtime (Home) does the passkey auth and holds the keys. The app runs no sign-in of its own and never holds a seed — it just adopts the identity the runtime projects in.

🏦 The runtime owns auth

Home authenticates with the passkey, derives & keeps the keys, and exposes whoami / sign / x25519_dh / ml_kem_decapsulate. One wallet identity, shared by every Hey app.

📦 The capsule owns nothing

No passkey ceremony, no recovery key, no seed in localStorage — there is no auth code in hey-pack at all. The app only adopts the projected did:key and asks the runtime to sign.

No runtime → no identity → the app does not open. Removing capsule-side auth also removes the XSS-exfiltration surface a stored seed would carry.

Two devices meet

The introduction — no plaintext, ever

Strangers bootstrap over an out-of-band invite link (shareable as a QR). Each side learns the other's keys + a private random queue.

The invite carries Alice's pubkeys, a random 256-bit queue, and a Double-Ratchet prekey. Bob's reply completes the ratchet. The link expires in 24h and is single-use; nothing readable ever touches the network.

A message, end to end

What you send vs. what Carrier carries

Alice types four characters. The relay forwards an opaque blob between two random pseudonyms.

Only Bob's device can derive the key (it needs both his X25519 private and his ML-KEM secret). The sender's identity rides inside the sealed box — "sealed sender" — so even the relay that routes the packet can't tell who sent it.

Security features

Closer to Signal + SimpleX than to "E2E chat"

Post-quantum hybrid

ML-KEM-768 (NIST FIPS 203) ⊕ X25519. Break both to read anything — defeats "harvest now, decrypt later."

Double Ratchet

Per-message chain keys + DH ratchet → forward secrecy and post-compromise security: a leaked key self-heals.

Sealed sender

Sender DID, signature & text live inside the ciphertext. The relay sees {dm.v2, envelope} and nothing else.

Metadata-blind routing

Per-pair random 256-bit queues + random pseudonyms. Your DID never appears in a topic name.

Length hiding

Plaintext padded to fixed buckets (256B→64K) before sealing — ciphertext leaks only the bucket, not the real size.

Incognito mode

Per-contact ephemeral identity — fresh Ed25519 + X25519 + ML-KEM keys, unlinkable to your real DID or your other contacts.

E2E attachments

Each file sealed with its own random key. Only ciphertext is stored; the key rides inside the sealed DM.

Reliable delivery

Failed sends queue in an outbox & retry with exponential backoff (5s→1h). DoS-hardened: bounded skips, no-panic parsing.

Threat model at a glance

What a network observer can & can't learn

🙈 Hidden from the relay

  • Message contents — PQ-hybrid AEAD ciphertext
  • Who sent it — sealed-sender; only a random pseudonym shows
  • Who it's for — recipient DID never in the topic
  • The real message size — bucket-padded
  • File contents & names — per-file E2E sealed
  • Your social graph — queues look like random traffic

👀 Still visible (honest caveats)

  • That a packet moved to some queue id
  • Network-layer node id / IP — needs a garlic overlay (future)
  • Approx. attachment size — file padding is a TODO
  • v1 ratchet FS/PCS is classical-only — PQ self-healing is reserved
  • Keys live in your runtime — wallet model: trust shifts to the node you (or your host) run, not the browser

The code says these out loud. Honesty about the boundary is part of taking the crypto seriously.
